Engineer Assessment Briefing

Lawpath Engineering - Full Stack Engineer

Executive Summary

You are being assessed for the Engineer position at Lawpath. This is a two-part assessment: a technical implementation challenge followed by a presentation. The assessment evaluates your ability to build modern web applications using Next.js with secure authentication, integrate with external APIs through GraphQL, implement proper data persistence and log-in with Elasticsearch, and communicate technical decisions effectively.

Key Outcome: Demonstrate you can build a production-ready Next.js application with secure authentication, address verification through API integration, data persistence with Elasticsearch, and then present your architectural decisions and scaling strategies.

Assessment Structure:

• Technical Implementation: 40% of total assessment

• Presentation & Architecture Discussion: 60% of total assessment

• Expected Time Investment: 1-2 days for implementation + presentation preparation

• Submission Deadline: 1 week from receipt

• AI Assistance: You are welcome to use AI tools to help complete this assessment

Description

Create an application using only Next.js with authentication and a Verifier form.

  • The application should require user authentication before accessing the Verifier form. Implement a simple login/registration system that stores user credentials in Elasticsearch or use a basic auth strategy.
  • The Verification form should capture a postcode, suburb, and state. Upon submission, the form must query the Australia Post API through a GraphQL proxy to confirm the validity of the entered address.

The application must persist the user's data across sessions. When the user closes and reopens the site, the form data should be restored.

Requirements:

Required Features:

  1. Implement a simple authentication system:
    • Users must login before accessing the Verification form
    • Support user registration and login
    • Store user credentials in Elasticsearch (create a users index) or implement a basic auth strategy
    • Maintain session state after login
  2. Implement the form using the latest features from Next.js.
  3. Implement the Verification form to validate postcode, suburb, and state.
  4. Integrate Apollo Client (or an equivalent library) to facilitate GraphQL interactions with the Australia Post API (See Appendix A) through a GraphQL proxy.
  5. Implement state persistence so that if a user closes the browser and returns later, the form data is restored.
  6. Nice to have (Bonus):
    • Store all verification attempts into Elasticsearch for log-in, including:
      • User ID or username of the logged-in user
      • Input data (postcode, suburb, state)
      • Timestamp of the attempt
      • Validation result (success/failure)
      • Error message if validation failed
    • Google Maps integration - display the location on a map when validation is successful.

Tests to Pass

A. Authentication - Demonstrate understanding of secure auth practices

  • Users cannot access the Verifier form without log-in in (protected routes)
  • Passwords are properly hashed (not stored in plain text)
  • Successful login redirects to the main application with proper session creation
  • User session persists across page refreshes (token/session management)
  • Logout functionality properly clears session/tokens
  • Handle auth errors gracefully (invalid credentials, session expiry)

B. Verification Form

  • Verify that the entered postcode aligns with the specified suburb. In the event of a mismatch, display an error message. E.g., "The postcode 2000 does not match the suburb Broadway".
  • Confirm that the inputted suburb is consistent with the selected state. If there's a discrepancy, display an error. For instance: "The suburb Ferntree Gully does not exist in the state Tasmania (TAS)".
  • If the postcode, suburb, and state align correctly, display the success message: "The postcode, suburb, and state input are valid."

C. See Appendix B for more examples to try.

D. Upon reloading the page, the user should remain logged in (if previously logged in), and the form data should be restored.

E. (Bonus): Ensure that all verification attempts are stored in Elasticsearch with the user ID for log-in purposes.
