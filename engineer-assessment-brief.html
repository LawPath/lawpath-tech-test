Executive Summary

You are being assessed for the Engineer position at Lawpath. This is a two-part assessment: a technical implementation challenge followed by a presentation. The assessment evaluates your ability to build modern web applications using Next.js, integrate with external APIs through GraphQL, implement proper data persistence and logging, and communicate technical decisions effectively.

Description

Create a single-page application using only Next.js with two interactive tabs: a Verifier tab and a Source tab

  • The Verifier tab should contain a form that captures a postcode, suburb, and state. Upon submission, the form must query the Australia Post API through a GraphQL proxy to confirm the validity of the entered address.
  • The Source tab should provide an interface to search for locations within a suburb or postcode. When the user enters a suburb/postcode, the application must query the same GraphQL proxy and display a selectable list of locations returned by the API into GoogleMaps.

The application must persist the user's selected tab and relevant data (form inputs, search results) across sessions. When the user closes and reopens the site, it should reopen on the last active tab with its data restored.

Requirements:

  1. Implement the form using the latest features from Next.js.
  2. Implement two switchable tabs within the same single-page application:
    • - Verifier tab with a form to validate postcode, suburb, and state.
    • - Source tab with a search field for suburbs/postcode and a list of matching results from the API.
  3. Integrate Apollo Client (or an equivalent library) to facilitate GraphQL interactions with the Australia Post API (See Appendix A) through a GraphQL proxy.
  4. Implement tab state persistence for both tabs so that:
    • - If a user closes the browser and returns later, the previously active tab remains selected.
    • - The data in that tab (search query/results) is restored.
  5. For the Source tab, we should be able to filter out locations based on their categories.
  6. For the Source tab, when a location is selected/clicked from the list of results, display the pinpointed selected location on Google maps.
  7. For the Verifier tab, display the pinpointed location of the input on Google maps.
  8. Store inputs and results from both the Verifier and Source tab into Elasticsearch for logging.
    • - Only store the logs of inputs and outputs for the selected location in the Source tab.
  9. Display the logs that were stored in Elasticsearch (requirement 8), you may proceed with the implementation as you see fit.

Tests to Pass

A. When using the Verifier tab

  • - Verify that the entered postcode aligns with the specified suburb. In the event of a mismatch, display an error message. E.g., "The postcode 2000 does not match the suburb Broadway".
  • - Confirm that the inputted suburb is consistent with the selected state. If there's a discrepancy, display an error. For instance: "The suburb Ferntree Gully does not exist in the state Tasmania (TAS)".
  • - If the postcode, suburb, and state align correctly, display the success message: "The postcode, suburb, and state input are valid."

B. See Appendix B for more examples to try.

C. When using the Source tab:

  • - If the inputted suburb name matches results from the API, display the list of matching suburbs.
  • - If there are no matching results, show a user-friendly message indicating that no matches were found (e.g., "No suburbs found for 'xyz'").

D. Upon reloading the page, the previously selected tab should be selected, and the previous search query and results (if any) should be restored.

E. Ensure that the inputs and single results from both the Verifier tab and the selected inputs/results from the Source tab (as per requirement 8) are stored in Elasticsearch.
